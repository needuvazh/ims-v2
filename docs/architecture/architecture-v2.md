# Architecture v2

## ASTI Integrated IMS Review Alignment

**Status:** Draft / Review-aligned  
**Scope:** Single-client ASTI IMS with branch-scoped data isolation  
**Architecture style:** TypeScript modular monolith with DDD package boundaries

## 1. Architecture Decision Summary

The reviewed architecture remains a modular monolith. This is the lowest-risk approach for ASTI because Enrollment, Finance, Attendance, Completion, Certificate, Corporate Training, and Documents are tightly coordinated business workflows.

The system shall not use microservices, event sourcing, CQRS, Redis, or an external broker in Phase 1 unless a later architecture decision record documents a concrete operational need.

## 2. Portal Model

The platform exposes four user-facing surfaces:

| Portal | Primary users | Core capabilities |
| --- | --- | --- |
| Admin Portal | Registrars, finance officers, HR/admin staff, branch managers, executives | Operations, RBAC, organization, CRM, admissions, finance, academics, reporting |
| Trainer Portal | Faculty and trainers | Assigned batches, timetable, attendance, completion recommendations |
| Student Portal | Students and sponsored learners with login access | Profile, timetable, attendance, fees, results, certificates |
| Public Verification | Employers and public verifiers | Certificate lookup by number or QR token |

Corporate clients are a major stakeholder group but are initially represented through Admin Portal workflows unless a separate corporate portal is approved.

## 3. Bounded Context Additions

The review adds or clarifies these bounded contexts:

| Context | Ownership | Integration policy |
| --- | --- | --- |
| Corporate Training | CorporateAccount, CorporateContactPerson, CorporateContract, CorporateProgram, CorporateParticipant | Uses Enrollment application services for learning lifecycle changes and Finance services for billing |
| Compliance and Documents | DocumentType, Document, DocumentVerification, expiry rules, compliance holds | Publishes expiry and verification events; does not bypass owning Student/Trainer status workflows |
| Biometric Integration | Terminal gateway registration, biometric event intake, sync checkpoint, idempotency keys | Anti-corruption layer; Attendance owns attendance records |
| Tally Integration | Financial export mapping, sync attempts, reconciliation state | Anti-corruption layer; Finance owns payments, receipts, refunds, invoices |

## 4. Integration Reliability

All external side effects shall use reliable integration patterns:

| Integration | Required pattern | Rationale |
| --- | --- | --- |
| Tally ERP | Transactional outbox, worker retry, reconciliation log | Prevents financial dual-write inconsistency |
| Payment gateway | Provider adapter plus outbox/webhook verification | Keeps vendor payloads outside domain models |
| WhatsApp/SMS/email | Notification template registry and delivery log | Preserves auditability and localization |
| Biometric terminals | Local gateway, SQLite buffer, idempotent cloud push | Prevents attendance loss during campus network outages |

Business transactions must complete against PostgreSQL first. External delivery failures must not roll back valid Enrollment, Attendance, or Finance state unless the business action explicitly depends on external authorization.

## 5. Branch-Scoped Data Isolation

ASTI uses a single-client database with logical branch separation. Operational tables shall carry `branchId` unless they are global configuration, audit, identity, or integration metadata records.

Every query path must resolve the active branch context from the authenticated session and enforce it server-side. UI filtering is not authorization.

## 6. Bilingual Architecture

The system shall support English LTR and Arabic RTL layouts. Localized business text shall use a consistent storage model:

```json
{
  "en": "Advanced Mechanical Diagnostics",
  "ar": "التشخيص الميكانيكي المتقدم"
}
```

Where Arabic search, sorting, reporting, or uniqueness requirements become material, use translation tables and indexes instead of opaque JSON-only storage.

## 7. Audit and Compliance

Finance, Attendance corrections, Completion approvals, Certificate issuance/revocation, Document verification, RBAC changes, and integration reconciliation state changes require append-only audit records.

Critical document expiry alerts shall be generated by a worker. Expiry may trigger a compliance issue or account hold according to ASTI policy.

## 8. Open Architecture Questions

1. Should corporate credit limit failures always block enrollment, or allow an approval override?
2. Do biometric terminals push to a local server, or must ASTI procure a local gateway device/client?
3. Is Tally expected to receive near-real-time entries, or is daily batch reconciliation acceptable?
4. Which localized fields require Arabic full-text search rather than display-only translation?
